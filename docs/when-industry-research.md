# When：业界调研

> 我们想做一个开源项目，叫 When。
>
> When 是一个通用的、独立部署的分布式延时投递组件，用 Java 写。它解决一件事：在指定的时间点，把一条消息精确投递到任意下游。下游可以是 Kafka、RocketMQ、HTTP Webhook、gRPC 服务、NATS，或者用户自己写的 Sink 插件。
>
> 这份文档是调研的整理。看完之后再决定怎么做、做多大、和现有方案怎么错开。

## 一、延时投递这件事在哪些地方出现

先确认一下需求侧。延时投递不是个抽象的概念，它在很多场景里都是必备能力。

电商和交易类最典型。订单创建后 30 分钟未支付要自动取消并释放库存、优惠券过期前几小时推送提醒、退款申请 24 小时未响应自动确认、拼团结束的精确时间点触发结算。这些场景几乎每个电商系统都有，而且每一个都是硬需求，不能用"定时跑批"之类的方式糊弄过去。

消息推送和通知类同样常见。定时推送、定时发布内容、不活跃用户唤醒、N 天后召回流程。任何 To C 产品都跑不掉，推送时机差一分钟和差一秒钟，对用户感知和业务转化的影响可能完全不同。

任务和工作流类里也大量出现。任务失败后延迟重试（指数退避）、报表定时生成、对账延时触发、审批超时自动流转。这些场景里的延时往往是系统稳定性设计的一部分，不是可选项。

风控和审计类也用。大额交易延迟到账给反洗钱系统留处理窗口、用户行为达到阈值后延迟触发规则引擎、交易数据延时归档。这类场景对精度和可靠性要求更高，漏一条或者晚触发可能有合规风险。

把这些场景放在一起看，共同点很清晰：业务系统需要在一个未来的精确时间点触发一个动作。这个动作可能是发一条 Kafka 消息、调一个 HTTP 接口、推一条通知、写一条数据。触发是延时的核心，投到哪里是延时的边界。

需求侧的真实状态：几乎所有有点规模的业务系统都需要延时能力。区别只是延时数量多少、精度要求高低、可靠性要求严不严。

## 二、消息中间件的延时能力到底支不支持

很多人第一反应是"延时不就是消息队列的事吗"。看下去就知道这件事远没有那么顺。

**Kafka** 完全没有原生延时能力。这是 Kafka 设计哲学的选择，Kafka 是为高吞吐流式数据设计的，延时不在它的核心场景里。社区里基于 Kafka 做延时都是外挂方案，要么用 Redis ZSet 加定时投递，要么用 Kafka 自身的内部 Topic 模拟时间分桶。两种做法都有自己的问题，没有一个真正干净的实现。

**RocketMQ** 在 4.x 版本只支持 18 个固定延时档位：1s、5s、10s、30s、1m、2m、3m、4m、5m、6m、7m、8m、9m、10m、20m、30m、1h、2h。不能任意秒级延时，最长上限 2 小时。这种"档位制"对简单场景够用，但凡是要"延时 17 分钟 23 秒"或者"延时 6 小时"这种业务，4.x 直接做不到。RocketMQ 5.0 才引入基于时间轮的任意延时能力，而且这个能力还在持续完善中。

**Pulsar** 原生支持任意延时。Delayed Message Tracker 这个机制做得相对干净，精度较好。但 Pulsar 整个体系比较重，Broker、BookKeeper、ZooKeeper 一整套，只为延时引入 Pulsar 对绝大多数场景都是杀鸡用牛刀。

**RabbitMQ** 本身不支持延时，社区有两种凑法。一种是 TTL + 死信队列，消息过期后被路由到死信队列再消费，缺点是粒度粗、不同延时需要不同队列、消息过期顺序和入队顺序不一致。另一种是官方的 Delayed Message Plugin，基于 Mnesia 数据库存延时消息到期再投递，缺点是大延时数量场景下性能撑不住。

把这四个放在一起看，共同问题非常清晰：这些方案的延时能力都和某个特定消息中间件绑死。要用 RocketMQ 的延时，前提是接受 RocketMQ 整套体系；要用 Pulsar 的延时，前提是接受 Pulsar 整套体系。

但业务侧真实需要延时投递的下游，远不止消息中间件。订单超时后可能要直接调一个 HTTP 接口取消订单、延时推送可能直接走 Push 服务、延时审计可能直接写一条 ClickHouse。这些下游和 Kafka / RocketMQ / Pulsar 没关系。消息中间件的内置延时能力，覆盖不了这些场景。

更现实的状态是：很多业务系统同时在用多种下游，Kafka 投一部分、HTTP 接口调一部分、内部 RPC 走一部分。如果延时能力绑定某个中间件，业务侧要么接受割裂（部分场景用 RocketMQ 延时，部分场景自己撸），要么硬把所有延时都套到一种中间件上（不合理）。这是消息中间件内置延时方案的根本边界。

## 三、SDK 类延时方案为什么不够

绕开消息中间件，另一类常见做法是基于 Redis 的延时队列 SDK。底层基本都是 Redis ZSet，以到期时间为 score 排序，客户端轮询拿到期消息。

**Redisson DelayedQueue** 是 Java 生态里用得最多的。基于 Redis 实现，API 简单，接入快。

**Go 生态有 HDT3213/delayqueue** 这类项目，GitHub Stars 大约 1000+，定位"简单、可靠、免安装的分布式延时消息队列"。架构上比 Redisson 略好一些，支持 worker 横向扩展、支持 Redis Cluster。

**Apache Curator** 也提供了 DistributedDelayQueue，基于 ZooKeeper 实现。理论上可用，但 ZK 本身就不是为大数据量设计的，延时消息数量稍微大一点性能就崩。

这一类方案的共同问题有几个层次，且彼此叠加。

最直接的是大 Key 问题。Redis ZSet 在元素数量大、score 跨度大的情况下，容易形成几十兆甚至几百兆的大 Key。Redis 单线程模型对大 Key 操作非常敏感，一次 ZRANGE 或者 ZREM 就可能卡住整个 Redis 实例几百毫秒。延时消息数量起来之后，这件事几乎必然踩到。

第二层问题是 SDK 形态本身的局限。延时逻辑都在客户端，客户端要轮询、要保证消费一次性、要处理重试、要维护本地状态。这意味着客户端版本升级困难，业务方升级周期长；可观测性靠业务方自己埋点，基础设施侧看不到全局数据；不同业务方接入版本不一致，出问题难定位；跨语言基本不可能，Java SDK 和 Go SDK 是两套实现，各自踩坑。

第三层问题是依赖业务自己的 Redis。SDK 模式下，延时消息的存储跑在业务方自己的 Redis 上。业务方有合理顾虑：延时上来的 QPS 会不会冲击业务正常的 Redis 使用、大 Key 会不会拖死整个 Redis 实例、运维责任怎么分。这些顾虑导致 SDK 方案推广困难，即使技术上能用，业务方也不愿意接。

SDK 类方案的天花板就在这里。它能解决"小规模延时"的问题，但解决不了"通用基础设施"的问题。

## 四、独立延时服务的尝试

绕开 SDK 形态，自然的下一步是做一个独立的延时服务。业务方通过 RPC 或 HTTP 提交延时请求，服务端负责存储、调度、到期投递。这种思路下，业界做过几次尝试，但都没能形成事实标准。

**Airbnb Dynein**（2019 年开源）是公开资料里能找到的、和"独立通用延时服务"定位最接近的开源项目。Dynein 的架构思路非常正，把延时投递的本质拆成"调度"和"队列"两件事，各自独立设计。服务本身无状态，跑在 Kubernetes 上。队列层用 AWS SQS，调度层用 DynamoDB 存调度表。

但 Dynein 的几个问题让它没能形成社区影响力。第一是强绑定云厂商，SQS 加 DynamoDB 是 AWS 的组合，不在 AWS 上的用户基本用不起来，这把潜在用户砍掉了一大半。第二是技术栈偏老，Java 加 Dropwizard 在 2019 年还算主流，放到现在已经显得有点重，新一代分布式中间件大多用 Go，Java 体系的运维和资源消耗都偏重。第三是开源后没有持续投入，GitHub 上 Dynein 的最后一次实质性更新已经几年前了，Issues 和 PR 基本无人回应。第四是产品形态不完整，没有 Web 管理台，只能靠 API 操作，作为开源项目让外部用户接受缺一道门面。

**Netflix Dyno-queues** 基于 Dynomite（Netflix 自己的 Redis 多区域版本）实现，强调多区域可用性，主要服务 Netflix 自身的多区域架构，通用性不强。Dynomite 本身这几年已经停止维护，Dyno-queues 跟着冷下来。

这两个案例说明了同一件事：独立延时服务这条路是对的，但大公司的尝试要么绑定自家基础设施，要么开源后没运营，要么产品形态不完整。真正能被广泛采纳的、通用的、社区活跃的独立延时服务，当下还没有。这件事的"位置"是空的。

## 五、任务队列和任务调度：边界要划清楚

讲完延时本身的方案，还需要划清楚另外两类容易混淆的方案。

**任务队列类：BullMQ、Celery、Sidekiq、Resque、Faktory**

这一类项目"延时"是附带功能，核心定位是"分发任务给 worker 异步执行"。BullMQ 在 Node.js 生态非常流行，Celery 是 Python 生态的标杆，Sidekiq 在 Ruby 生态接近事实标准。它们都支持给任务设置延时（BullMQ 的 `delay` 参数、Celery 的 `ETA/countdown`、Sidekiq 的 `perform_in`）。

但任务队列和延时投递是两件事。任务队列的核心动作是"把任务交给 worker"，worker 拿到任务后在自己的进程里执行业务逻辑，延时是这个流程里的一个可选项。延时投递的核心动作是"在指定时间把一条消息送到下游"，下游处理什么是下游的事，延时组件不关心、也不需要 worker 概念。

两个场景有交集，比如"3 分钟后给用户发邮件"，可以用任务队列（把"发邮件"作为任务延时执行），也可以用延时投递（在 3 分钟后投递一条消息给邮件服务）。但两者的设计重心不同：任务队列围绕"任务执行"展开（concurrency 控制、worker 池、任务依赖编排、rate limiting），延时投递围绕"精确触发"展开（延时精度、调度规模、下游灵活性、可靠投递）。用任务队列代替延时投递，会得到一个不专注的方案。

**任务调度类：XXL-JOB、PowerJob、ElasticJob、Quartz**

XXL-JOB 在国内开源生态影响力非常大，GitHub Stars 接近 3 万，是国内分布式中间件领域少有的全民认知项目。Web 管理台做得很好，执行器机制成熟，接入简单。

但 XXL-JOB 的核心是 cron 风格的任务调度，管理员预先定义任务、配置调度规则（cron 表达式或固定间隔）、注册执行器，调度中心按规则触发。这和延时投递的需求模型完全不同。延时投递的典型用法是：业务代码运行时凭空产生一条延时消息，扔给延时组件，30 分钟后到期投递出去。每条延时消息可能都不一样，不同的目标、不同的延时时长、不同的内容、不同的业务上下文。这种用法在 XXL-JOB 里很别扭，你不会为每个待取消的订单创建一条新的调度 Job。

把这两类一起看，任务队列和任务调度都不能填"通用延时投递"这个位置。它们各自有清晰的领地，但延时投递这件事它们都不专注。

## 六、不同语言生态在这件事上的状态

不同语言生态里这个领域的状态差异很大，值得单独梳理一下。

**Node.js** 生态有 BullMQ。任务队列加延时一体，生态完整，文档好，多语言客户端（已经扩展到 Bun、Python、Elixir、PHP），社区活跃。Node.js 工程师做"任务异步执行加延时"基本不用选，BullMQ 就是答案。

**Python** 生态有 Celery。十几年的项目，生态成熟，虽然定位偏任务队列但延时能用。

**Ruby** 生态有 Sidekiq。一个人撑了十几年的项目，免费版加 Pro 加 Enterprise 的分层做得很成熟。Ruby 工程师做异步任务基本默认用它。

**Go** 生态在这件事上基本空着。HDT3213/delayqueue 是 SDK 形态的延时队列，Stars 还可以，但定位是库不是服务。其他能搜到的 Go 延时项目大多是个人 demo 级实现，没有进入广泛使用。

**Java** 生态的状态比较特别。一方面 Java 是企业级后端最大的语言，需要延时投递的场景密度最高；另一方面 Java 里和延时沾边的项目最多，但没有一个专注做"通用延时投递"。Quartz 定位是任务调度框架，核心是 cron 表达式加 Job 执行，延时不是它的本职。Redisson 的 DelayedQueue 是 SDK 形态，大 Key、可观测、依赖业务 Redis 这几层局限前面已经讲过。XXL-JOB 是任务调度平台，29.6k stars 证明 Java 生态对这类基础设施的需求强度，但它解决的是 cron 风格的调度问题，不是延时投递。

Java 生态里没有一个项目正面回答"业务方需要在指定时间把消息精确投递到任意下游"这个问题。通用延时投递这件事散落在不同项目里，没有人把它做完整。

## 七、综合判断

把前面的内容收一下。

**延时投递是个真实存在的需求。** 订单、推送、任务、风控、审计，业务系统几乎跑不掉。需求的规模、精度、可靠性要求虽有不同，但需求本身是稳定的、广泛的、长期的。

**业界凑出来的方案多，真正合适的少。** 消息中间件的内置延时绑定自己的体系，Kafka 没有、RocketMQ 4.x 受限、Pulsar 太重、RabbitMQ 靠 hack。SDK 类方案在大 Key、可观测、依赖业务 Redis 这些点上踩坑，长期不可持续。

**独立延时服务这条路被尝试过，但没有活下来的。** Airbnb Dynein 是最接近的尝试，架构思路是对的，但绑定云厂商、技术栈偏老、产品形态不完整、社区运营跟不上，几年下来基本停滞。Netflix Dyno-queues 强绑定自家基础设施，通用性不强。这件事的"位置"目前是空的。

**任务队列和任务调度不能代替延时投递。** BullMQ、Celery、Sidekiq 的核心是 worker 执行，延时是附带。XXL-JOB 的核心是 cron 风格调度，场景不匹配。它们各自在自己的领域是事实标准，但不是延时投递的事实标准。

**不同语言生态在这件事上的状态分化明显。** Node.js 有 BullMQ、Python 有 Celery、Ruby 有 Sidekiq，但都偏任务队列定位。Java 生态需求最密集，但没有专注做"通用延时投递"的项目。Go 生态基本空着。

把这些放在一起，整个领域的当下状态大致可以这样描述：需求是真实且广泛的，方案是分散且各有局限的，通用且独立且专注于延时投递的开源项目当下没有事实标准。

调研到此为止。具体怎么填这个位置、做什么功能、和现有方案怎么错开，是 When 后续需求文档要回答的事。
